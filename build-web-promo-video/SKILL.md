# Build Web Promo Video

Produce one complete local video bundle through a modular pipeline. Keep creative decisions in the
agent, keep media generation deterministic where possible, and verify both media parameters and
visual output before delivery.

## Workflow

### 1. Establish scope and evidence

Identify the project root, intended audience, target platforms, desired duration, core message,
real visual states, public links, and licensed media. Inspect the project, current screenshots,
runtime commands, and existing videos before writing a storyboard.

Read [references/storyboard-standard.md](references/storyboard-standard.md) before choosing scenes.
Read [references/media-rights.md](references/media-rights.md) before using narration services,
music, fonts, logos, screenshots, or supplied clips. Do not invent user counts, performance results,
awards, testimonials, or product states.

### 2. Create and review the configuration

Copy `assets/config-template/video-project.json` into the project and customize it. Read
[references/config-schema.md](references/config-schema.md) for every field. Keep scene IDs stable and
use real page paths, public URLs, or explicitly supplied clip paths.

Choose one capture source per scene:

- `path`: serve and record a page below the configured project root.
- `url`: record an already running/public web page.
- `clip`: use an existing local video file without browser capture.

Keep `capture.perProfile` enabled for responsive pages that need both landscape and vertical
output. Disable it only when reusing one capture is intentional, such as supplied clips or a
single-layout page whose `contain`/`cover` result has been visually approved.

Prefer declarative actions for clicks, keys, typing, mouse movement, scrolling, and waits. Use a
reviewed JavaScript adapter only when the page needs project-specific setup or debug hooks. Read
[references/capture-adapters.md](references/capture-adapters.md) before adding adapter code.

### 3. Inspect without rendering

Run:

```text
node <skill>/scripts/inspect_project.mjs --config <video-project.json>
```

Review `project-facts.json`, `storyboard.md`, and `timeline.json`. Confirm scene order, total duration,
viewport, narration timing, output profiles, file existence, and adapter paths. Resolve all blockers
before recording.

### 4. Prepare the media environment

Read [references/environment.md](references/environment.md). Install the script dependencies and a
full FFmpeg/FFprobe pair. Use `FFMPEG_PATH` and `FFPROBE_PATH` when the tools are not on `PATH`.

Use an installed Playwright browser or set `PLAYWRIGHT_CHANNEL=msedge`/`chrome`. Install the optional
Python `edge-tts` dependency only when the configuration selects that provider. Never request or
copy browser cookies, login profiles, passwords, or platform tokens.

### 5. Build the complete bundle

Run the default full pipeline:

```text
node <skill>/scripts/build_promo_video.mjs \
  --config <video-project.json> --force
```

The orchestrator must:

1. inspect the project and create the storyboard/timeline;
2. capture configured pages or copy supplied clips;
3. generate/copy narration when enabled;
4. generate procedural music or copy supplied audio;
5. create SRT and profile-specific ASS subtitles;
6. render configured landscape/vertical clean and captioned MP4 files;
7. create contact sheets for visual review;
8. run ffprobe validation and write JSON/Markdown reports.

Use `--force` only for an output directory already marked as generated by this Skill. Never delete
an unmarked directory. Use `--skip-capture` only when a valid recordings manifest already exists.

### 6. Validate and visually inspect

Run the validator independently after any manual media change:

```text
node <skill>/scripts/validate_video.mjs --config <video-project.json>
```

Resolve every blocker. Review warnings instead of suppressing them. Inspect every contact sheet and
at least one complete clean/captioned render. Parameter validation cannot detect a blank Canvas,
wrong app state, unreadable subtitle, poor crop, narration mismatch, weak pacing, or misleading
scene order.

### 7. Deliver the complete bundle

Deliver:

- `project-facts.json`, `storyboard.md`, and `timeline.json`;
- `recordings/` and its capture manifest;
- `narration/`, `audio/`, and `subtitles/`;
- `videos/` with all configured clean/captioned profiles;
- `review/` contact sheets;
- `editable-manifest.json` listing reusable clips, audio, and subtitle files;
- `validation-report.json` and `validation-report.md`.

State which scenes were browser-recorded, adapter-driven, or supplied as clips. Report exact codec,
dimensions, frame rate, duration, audio format, file size, and visual-review status. Do not claim a
successful video from an FFmpeg exit code alone.

## Supported modes

- **Full**: browser capture + narration + music + subtitles + all renders + validation.
- **Existing clips**: skip browser capture for scenes with `clip` sources.
- **Silent narration**: use subtitle segments with narration provider `none`.
- **User audio**: use narration provider `files` and/or audio mode `file`.
- **Optional TTS**: use narration provider `edge-tts` after reviewing service and redistribution
  terms.

All modes remain one Skill and one output contract.

## Non-negotiable rules

- Do not upload or publish videos to Bilibili, YouTube, social platforms, or cloud storage.
- Do not capture authenticated/private pages without explicit authorization.
- Do not read or distribute browser profiles, cookies, tokens, or credentials.
- Do not include third-party music, fonts, logos, voice output, or clips without usable rights.
- Do not present generated mockups as real runtime footage.
- Do not make empty/black frames pass merely because codec checks succeed.
- Do not overwrite an unmarked output directory or unrelated user files.
- Do not bundle FFmpeg binaries, browser binaries, fonts, generated voice, or large demo videos in
  the Skill.
- Do not hide service/network requirements for optional TTS.
